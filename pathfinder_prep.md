## 1. このプロジェクトは何か

**パスファインダー（Pathfinder）**は、医学論文のエビデンスを「自分の地図」として構造化するためのウェブアプリです。

開発者 Yujin が既に公開している姉妹アプリ **Citrail ―引用を辿る―**（引用文献ネットワーク可視化アプリ、GitHub Pages で公開済み、別リポジトリ）と連携して使うことを想定していますが、独立したアプリとして開発します。

### 背景と目的

- Citrail は 1 本の論文を起点に引用・被引用ネットワークを**網羅的**に表示するが、そこには臨床疑問と無関係な論文も混じる（例：抗菌薬の効果を知りたいのに、統計手法の論文が引用に含まれる）。
- 論文を読む最終目的は、エビデンスを統合して**自分の知見を持つ**こと。
- そこで、Citrail で集めた「足跡」の中から臨床的に意味のある論文を**人間が選び、意味づけしてつなぎ直す**ためのツールがパスファインダー。

### 両アプリの対比（コンセプトの要）

| | Citrail | パスファインダー |
|---|---|---|
| 探索の性格 | 網羅的・受動的 | 選択的・能動的 |
| 図の意味 | 事実としての引用関係 | 自分で作る「エビデンスの地図」 |

### 思想の核・独自性（最重要）

**「事実の関係」と「人間の意味づけ」を、区別しながら同じ地図に重ねられること。**

- 自動リンク：ノードに DOI を入れると論文情報を自動取得し、DOI 入りノード同士は OpenAlex の引用・被引用関係に基づいて自動で線が引かれる（＝事実）。
- 手動リンク：ユーザーが自分の解釈でノード間に線を引ける（＝意味づけ）。
- 両者は見た目で区別できるように表示する（区別の方法は未決定・要検討）。

この二層構造が、単なる引用図エディタとの違いです。設計判断に迷ったら、この思想に立ち返ってください。

## 2. 決定済み事項

- 土台のイメージは、Yujin が以前 Claude と作った**ループ図アプリ**（ノード・矢印・メモ・ラベルを手動で自由配置できるエディタ）。これに論文機能を足す。
- ノードに DOI を入力すると論文情報を自動取得。複数の DOI 入りノード間は OpenAlex で引用関係を自動判定して線引き。
- 手動リンク機能あり。自動／手動の線は視覚的に区別する。
- まっさらな状態からも開始できるが、Citrail からのインポート開始も可能にする：Citrail 側に「パスファインダーを開始」ボタン → ノードを複数選択 → 選んだノードが配置済みの状態でパスファインダーが起動、という連携を想定。
- Citrail とは別リポジトリ・別アプリとして開発（大規模開発になるため独立プロジェクト化した）。

## 4. 技術的な前提・参考情報

パスファインダー自体の技術構成は論点 4 で決めますが、姉妹アプリ Citrail の構成が参考になります：

- 静的サイト（HTML/JS/CSS のみ、ビルド不要）を GitHub Pages で公開
- データソースは OpenAlex API（利用者自身の API キーを登録する方式）、研究種別判定に PubMed（NCBI E-utilities）も使用
- 描画は Cytoscape.js（vendor/ にセルフホスト）
- お気に入り等の保存は Local Storage
- サーバー・アカウント機能・個人情報収集なし（公開リスクを構造的に回避する方針）

同じ方針（静的サイト＋GitHub Pages）を踏襲するのが自然ですが、パスファインダーは「編集して保存するエディタ」なので、保存・共有の設計は Citrail より重要になります（論点 4）。

## 7. 論点1〜4の決定事項（2026-07-23 セッションで確定）

4つの論点すべてについて、Yujinさんとの議論で仕様が固まりました。次回セッションはここから実装計画の相談、または実装そのものに進んでください。

### 論点1：地図の構成要素の仕様

**ノードの種類**（固定4種類。ユーザーが種類を自由追加する仕組みは不要）

| 種類 | 役割 | 個数制限 | 中身 |
|---|---|---|---|
| 論文ノード | 事実 | 複数可 | DOIは**任意入力**（空欄可）。DOI入力時はOpenAlexから書誌情報を自動取得し自動リンク対象になる。DOIがない場合（ガイドライン・教科書など）はタイトル等を手入力。加えて**自由記述メモ欄**（鍵となる知見・引用文など）を持つ |
| 臨床疑問ノード | 思考の起点 | **地図に1つだけ** | （内容の詳細仕様は未検討） |
| 自分の結論ノード | 思考の到達点 | **地図に1つだけ** | （内容の詳細仕様は未検討） |
| メモノート | 補助的な注釈 | 複数可 | 他ノードとリンクで接続可能（付箋止まりにしない） |

**手動リンク（人間の意味づけの線）**
- 種類：定型カテゴリ（支持する／反論する／補足する など）＋自由記述ラベルの併用
- 方向：一方向／双方向／無方向を線ごとに選択可能
- **同じノードペアに自動リンクと手動リンクが共存できる**（「事実」と「意味づけ」を重ねるという思想の核に合致）

### 論点2：自動リンクと手動リンクの視覚的区別

- **自動リンク（事実）＝点線・グレー（固定・中立色）**
- **手動リンク（意味づけ）＝実線・色はプリセットパレットから選択**（支持／反論／補足などのカテゴリに、用意された色候補からユーザーが割り当てる。緑＝支持／赤＝反論のような固定割当ではない）
- 決定理由：パスファインダーは「事実の記録」ではなく「自分の知見を固める」アプリなので、確定的な意味づけである手動リンクこそ実線（強い表現）、参考情報にすぎない自動リンクは点線（弱い表現）とする
- 同一ノードペアに自動＋手動が共存する場合：2本の線を少しカーブさせて並べて描画し、常に両方が見える状態にする

### 論点3：Citrailとの連携仕様

事前調査で判明した事実：このリポジトリ（`citation-history-map`）自体がCitrail本体であり、`https://yujin0722-lgtm.github.io/citation-history-map/` として公開される設計（`docs/publish-guide.md`）。パスファインダーも同じGitHubアカウントで公開すれば `https://yujin0722-lgtm.github.io/pathfinder/` となり、**両者は同一オリジン**（`yujin0722-lgtm.github.io`）になるため、Local Storageを共有できる。

- **選択の仕組み**：Citrailの既存「お気に入り」機能とは**別に**、ネットワーク図上にその場限りの「選択」チェックボックスを新設する（お気に入りは「後で読み返す用」、こちらは「今回の臨床疑問に関連する論文を選ぶ用」で目的が異なるため）
- **受け渡し方式**：同一オリジンのLocal Storage経由（URL埋め込みは不使用。サーバー不要でCitrailの「サーバー・アカウントなし」方針を維持できる）
  - キー名：`chm_to_pathfinder`
  - データ形式：選択した論文の配列（`id, title, authors, year, journal, doi, pmid`）のJSON
  - パスファインダー側は読み込み後にこのキーを消費（削除）する
- **受け取り時の挙動**：パスファインダーに未保存の作業中の地図がある状態で新規の論文が届いた場合、「新規に始めるか、今の地図に追加するか」を毎回確認する

### 論点4：技術構成と保存・共有

ループ図アプリ（`yujin0722-lgtm/loop-diagram-app`）を事前調査した結果、**Supabase（PostgreSQL＋認証＋リアルタイム通信）によるクラウド保存・メールログイン・リンク共有・共同編集機能**が組み込まれていることが判明。これはCitrailの「サーバー・アカウント機能・個人情報収集なし」という方針と矛盾するため、パスファインダーではこの部分は流用しないことで合意。

- **技術構成**：単一HTMLファイル（HTML/CSS/JS、ビルド不要）、GitHub Pages公開。描画はループ図アプリと同じSVG手描画方式を流用（外部ライブラリ不使用）。**Supabase関連のコード（ログイン・クラウド保存・リアルタイム共同編集）は持ち込まない**
- **保存**：サーバー・アカウントなし。「1地図＝1臨床疑問」という構造のため、Local Storageに複数地図を一覧管理する「地図一覧」機能を新設。編集のたびに**自動保存**（手動保存ボタンは不要）
- **共有**：JSONファイルの書き出し／読み込みで手渡し（ループ図アプリのJSON書き出し機能をベースに流用）

## 8. 論点1の補足決定（2026-07-23 セッション、続き）

### 臨床疑問ノード・自分の結論ノードの中身

- 臨床疑問ノード：**自由記述のみ**（PICO形式などの構造化入力は導入しない）
- 自分の結論ノード：自由記述

### ノードの見た目（最終版）

| 種類 | 形 | 色 |
|---|---|---|
| 論文ノード | **研究種別によって変化**（Citrailの分類ロジックをそのまま流用：PMIDがあればPubMedの公式Publication Typeで判定、なければタイトルからの暫定判定）。RCT＝ひし形／メタ解析・SR＝六角形／レビュー＝角丸長方形／観察研究＝三角形／ガイドライン＝星形／症例報告＝タグ形／その他・未判定＝楕円 | 白（固定） |
| 臨床疑問ノード | 二重枠の長方形 | 水色 |
| 自分の結論ノード | 二重枠の長方形 | 緑 |
| メモノート | 付箋風（角が折れた四角） | 黄色 |

補足：当初「臨床疑問＝六角形」を検討したが、論文ノードの「メタ解析・SR＝六角形」と衝突するため、臨床疑問・自分の結論は論文が使わない「二重枠の長方形」に変更した（両者は色（水色／緑）で区別）。
