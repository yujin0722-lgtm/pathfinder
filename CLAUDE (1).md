## 1. このプロジェクトは何か

**パスファインダー（Pathfinder）**は、医学論文のエビデンスを「自分の地図」として構造化するためのウェブアプリです。

開発者 Yujin が既に公開している姉妹アプリ **Citrail ―引用を辿る―**（引用文献ネットワーク可視化アプリ、GitHub Pages で公開済み、別リポジトリ）と連携して使うことを想定していますが、独立したアプリとして開発します。

### 背景と目的

- Citrail は 1 本の論文を起点に引用・被引用ネットワークを**網羅的**に表示するが、そこには臨床疑問と無関係な論文も混じる（例：抗菌薬の効果を知りたいのに、統計手法の論文が引用に含まれる）。
- 論文を読む最終目的は、エビデンスを統合して**自分の知見を持つ**こと。
- そこで、Citrail で集めた「足跡」の中から臨床的に意味のある論文を**人間が選び、意味づけしてつなぎ直す**ためのツールがパスファインダー。

### 両アプリの対比（コンセプトの要）

| | Citrail | パスファインダー |
|---|---|---|
| 探索の性格 | 網羅的・受動的 | 選択的・能動的 |
| 図の意味 | 事実としての引用関係 | 自分で作る「エビデンスの地図」 |

### 思想の核・独自性（最重要）

**「事実の関係」と「人間の意味づけ」を、区別しながら同じ地図に重ねられること。**

- 自動リンク：ノードに DOI を入れると論文情報を自動取得し、DOI 入りノード同士は OpenAlex の引用・被引用関係に基づいて自動で線が引かれる（＝事実）。
- 手動リンク：ユーザーが自分の解釈でノード間に線を引ける（＝意味づけ）。
- 両者は見た目で区別できるように表示する（区別の方法は未決定・要検討）。

この二層構造が、単なる引用図エディタとの違いです。設計判断に迷ったら、この思想に立ち返ってください。

## 2. 決定済み事項

- 土台のイメージは、Yujin が以前 Claude と作った**ループ図アプリ**（ノード・矢印・メモ・ラベルを手動で自由配置できるエディタ）。これに論文機能を足す。
- ノードに DOI を入力すると論文情報を自動取得。複数の DOI 入りノード間は OpenAlex で引用関係を自動判定して線引き。
- 手動リンク機能あり。自動／手動の線は視覚的に区別する。
- まっさらな状態からも開始できるが、Citrail からのインポート開始も可能にする：Citrail 側に「パスファインダーを開始」ボタン → ノードを複数選択 → 選んだノードが配置済みの状態でパスファインダーが起動、という連携を想定。
- Citrail とは別リポジトリ・別アプリとして開発（大規模開発になるため独立プロジェクト化した）。

## 3. 未検討の論点（ここから議論を再開する）

チャットでは次の 4 論点を挙げ、**論点 1 から議論を始める**ことで合意した直後に引き継ぎとなりました。

1. **地図の構成要素の仕様**
   - ノードの種類：論文ノードだけか、「臨床疑問」「自分の結論」のような論文でないノードも置けるか（←アプリの性格を決める最重要論点。単なる引用図エディタか、エビデンス統合の思考ツールかの分かれ目）
   - 手動リンクに何を書けるか（ラベル、方向、種類など）
2. **自動リンクと手動リンクの視覚的区別**（線種・色など）
3. **Citrail との連携仕様**（インポートの形式＝JSON 等、受け渡し方法）
4. **技術構成と保存・共有**（ループ図アプリの流用範囲、保存方法）

## 4. 技術的な前提・参考情報

パスファインダー自体の技術構成は論点 4 で決めますが、姉妹アプリ Citrail の構成が参考になります：

- 静的サイト（HTML/JS/CSS のみ、ビルド不要）を GitHub Pages で公開
- データソースは OpenAlex API（利用者自身の API キーを登録する方式）、研究種別判定に PubMed（NCBI E-utilities）も使用
- 描画は Cytoscape.js（vendor/ にセルフホスト）
- お気に入り等の保存は Local Storage
- サーバー・アカウント機能・個人情報収集なし（公開リスクを構造的に回避する方針）

同じ方針（静的サイト＋GitHub Pages）を踏襲するのが自然ですが、パスファインダーは「編集して保存するエディタ」なので、保存・共有の設計は Citrail より重要になります（論点 4）。

- Citrail での協働スタイル：小さいバージョン刻みで実装 → Yujin が実際に動かして動作確認とフィードバック → 修正、の反復。細かい改修要望は TODO として溜め、まとまったところで一括改修することもあります。同じスタイルを踏襲してください。
